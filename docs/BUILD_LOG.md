# Build Log

A chronological record of what was built, in what order, and **why** each
non-obvious decision was made. Newest entries at the bottom. Kept so a
future session (human or AI) can reconstruct the reasoning, not just the code.

## 2026-07-05 — Project scaffold

- **Scaffolded with `create-next-app@16.2.10`** (Next.js 16.2.10, React 19,
  Tailwind v4, TypeScript, ESLint, App Router, `src/` dir, `@/*` alias).
- **Gotcha hit immediately:** the repo folder is `smallBizzBoiler` (capital
  letters), and npm forbids capitals in package names, so `create-next-app .`
  refused to run. **Fix:** scaffolded into a temp dir as
  `smallbizz-boilerplate` and copied the files in. The `package.json` name
  stays `smallbizz-boilerplate` — it's never user-visible.
- The pre-existing git repo (one README commit, remote already on GitHub)
  was kept; the scaffold's own `.git` was discarded.
- Deleted the scaffold's demo SVGs and its stub `AGENTS.md`/`CLAUDE.md`
  (replaced by our real handoff doc at the end).

## 2026-07-05 — Consulted Codex on static-export pitfalls

Asked GPT (Codex CLI) four targeted questions before writing code. Answers
that shaped the build:

1. `app/sitemap.ts` / `app/robots.ts` **do** emit `/sitemap.xml` and
   `/robots.txt` under `output: 'export'`, no `force-static` needed (they're
   static unless they touch request-time APIs).
2. GA4 via `next/script strategy="afterInteractive"` works fine in a static
   export; single page means no route-change tracking concerns.
3. `trailingSlash: true` recommended for Cloudflare Pages (routes emit as
   `route/index.html`) — adopted in `next.config.ts`.
4. Web3Forms is CORS-friendly from static pages; use a client component,
   plain `fetch` POST, no custom headers.

## 2026-07-05 — Core architecture decisions

- **`output: 'export'` + `images.unoptimized` + `trailingSlash`** set in
  `next.config.ts` first, so every subsequent decision was made under
  static-export constraints instead of retrofitting them.
- **Single source of truth:** all content/theming in `src/lib/site.config.ts`,
  typed by `src/lib/types.ts`. Components import the config directly rather
  than receiving props — with exactly one data source, prop-drilling would
  add ceremony without adding flexibility.
- **Theming via CSS variables, not tailwind.config:** Tailwind v4 has no JS
  config by default. `layout.tsx` writes `--primary` / `--accent` /
  `--site-font` inline on `<html>` from the config; a `@theme inline` block
  in `globals.css` registers them as Tailwind tokens. This keeps the
  "restyle = edit two hex codes" promise and even makes opacity modifiers
  (`bg-accent/10`) work against config-supplied colors.
- **Fonts:** `theme.fontFamily` is a plain CSS font stack (default: system
  fonts) instead of `next/font`. Reason: `next/font` requires a static
  import per font, which would break "font is just config data". Custom
  Google fonts are documented in the README as an opt-in step.
- **Opening hours modeled as `{days: Weekday[], open, close}`** with full
  schema.org day names — the exact strings JSON-LD needs — and display
  formatting (e.g. "Tue – Thu", "11:00 AM") done by helpers in
  `src/lib/format.ts`. One data shape feeds both the visible hours table
  and the structured data.
- **Prices are display strings** (`"$4.25"`, could be `"from $89"`), because
  salon/mechanic pricing is often not a clean number. JSON-LD extracts the
  numeric part via `priceNumber()`; `business.currency` (default USD)
  supplies the currency code.

## 2026-07-05 — Components & SEO

- Built `Hero`, `SpecialsBanner`, `ServiceMenu`, `About`, `Contact`,
  `ContactForm`, `Footer`, `JsonLd`. All server components except
  `ContactForm` (`"use client"` — the only interactive island), so the page
  ships minimal JS.
- **SpecialsBanner returns `null`** when no specials are active — clients
  without promos get a clean page with zero config gymnastics.
- **Specials render twice by design** (slim top banner + highlighted cards
  atop the menu): the banner catches attention, the cards carry the detail.
- **JSON-LD branches on businessType:** `Restaurant` emits `hasMenu`
  (Menu → MenuSection → MenuItem — Google's restaurant format); all other
  types emit `hasOfferCatalog` with `Service` items. Same config data,
  the schema shape Google prefers for each vertical.
- **Hero uses a brand-gradient, not a photo** — looks finished with zero
  client assets; swapping in a photo is a one-class change documented in
  the component.
- **OG placeholder:** no ImageMagick on this machine, so `public/og.png`
  (1200×630 brand gradient) was generated by a small Node script writing
  raw PNG chunks (`zlib.crc32` needs Node ≥22 — this machine has 25).
  It's a placeholder; each client should get a real photo.
- Demo business is a fictional taqueria ("El Camino Taqueria", Worcester MA,
  555 phone number) — realistic enough to demo, obviously fictional enough
  to never collide with a real business.

## 2026-07-05 — First build, one failure, verification

- **Build failed on first run:** Next 16.2 refuses `robots.ts`/`sitemap.ts`
  under `output: 'export'` without `export const dynamic = "force-static"`
  (contradicting both the online docs and Codex's earlier answer). Added the
  export to both files → build green. Recorded as a GOTCHA in CLAUDE.md.
- **Verified the export, both env directions:**
  - Empty env: builds; robots/sitemap use the example.com placeholder; NO
    `gtag` anywhere in the HTML; JSON-LD present with `@type: Restaurant`
    + `hasMenu`; theme CSS vars inlined on `<html>`; OG tags present.
  - Vars set (`SITE_URL`, `GA_ID`, `WEB3FORMS_KEY`): GA script baked in,
    real domain in robots/sitemap/og:image. Graceful degradation works
    both ways. ESLint clean.

## 2026-07-05 — Codex code-review pass, 4 findings fixed

Asked Codex (GPT) to review `src/` for real defects. All four findings were
legitimate and fixed:

1. **ContactForm had no pre-hydration fallback** — submitting before React
   loads would do a useless page reload. Fix: `action="https://api.web3forms.com/submit"
   method="POST"` on the `<form>` plus hidden `access_key`/`subject` inputs,
   so a plain browser POST works too; the JS path now reads those same
   hidden fields instead of appending duplicates.
2. **JSON-LD injection could theoretically be broken by a `</script>` in a
   config string.** Fix: escape `<` as `<` in the stringified JSON.
3. **Hero tagline in accent-on-primary was ~3.4:1 contrast** (fails WCAG AA
   for small text). Fix: tagline is now `text-white/80`; the accent color is
   reserved for surfaces where dark text sits on it (CTA button, banner).
4. **`next start` script is wrong for a static export** (there's no server
   to start). Replaced with `npm run preview` → `npx serve out`.

Final re-build + lint after the fixes: green.
